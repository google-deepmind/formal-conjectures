# Statistics / Analytics page — implementation plan

> Draft, not for commit. Add `site/STATS_PLAN.md` to `.gitignore` (or keep
> untracked) until we're ready to land any of this.

## 0. Goal in one sentence

Add a `/stats/` page to the Formal Conjectures website that surfaces
**derived structure** — cross-tabulations, ratios, time-evolution, and
health metrics — built from data the repo already maintains
(`@[category]`, `@[AMS]`, `@[formal_proof]`, source-collection module
prefix, git history). No new metadata systems.

## 1. Where things live today

The website is a **pure-static, zero-dep build** — relevant for what we
can do without adding tooling.

| Concern | File |
|---|---|
| Raw data input (per-theorem) | `site/data/conjectures.json` (produced by `lake exe extract_names`; CI-built; gitignored) |
| Per-theorem schema | `theorem`, `module`, `category`, `subjects[]`, `formalProofKind`, `formalProofLink`, `hasSorryFreeProof`, `statement`, `docstring` (see `scripts/extract_names.lean:65-83`) |
| Derived JSON consumed by client | written by `site/build.js:341-344` as `{ conjectures, stats, amsSubjects, versoFragments }` |
| Existing aggregates | `computeStats` in `site/build.js:200-219` — `byCategory`, `byCollection`, `bySubject` (1-D counts only) |
| Templates | `site/src/templates/{index,browse,theorem,contribute,about}.html` |
| Styles | `site/src/css/style.css` |
| Client JS | `site/src/js/{main,browse,theorem,voting}.js` |
| Build entrypoint | `site/build.js` (Node, no deps); `npm run build` |
| Nav links | duplicated in every template (`<ul class="site-nav__links">`) |
| Source collection map | `SOURCE_COLLECTIONS` in `site/build.js:91-105` |
| AMS subject map | `AMS_SUBJECTS` in `site/build.js:22-86` |

Key constraints to respect:

- **No external Node deps.** `package.json` has zero deps and the build
  is intentionally minimal. Anything new should stay vanilla Node + DOM.
- **Static output only.** No backend; everything is precomputed at build
  time and served as JSON + HTML.
- **`conjectures.json` is generated by CI.** Locally it may not exist —
  build already handles this. Stats build must degrade gracefully too.
- **Module → collection → subject already standardized** via the maps
  above. Reuse them; do not re-derive.

## 2. Scope split (suggested PR sequence)

The user asked for a small, visible PR-1 first, then incremental
follow-ups. Concrete sequence:

### PR 1 — Subject × status cross-tab (the "obviously missing" view)

**Deliverable:** a new page `/stats/` with one section: a table where
rows = AMS subjects, columns = categories
(`research open`, `research solved`, `textbook`, `test`, `API`),
plus a `formal_proof` column and a row total. Each cell is a link to the
matching `/browse/?...` filter so users can drill in.

**Why this first:**
- Single new computation (`bySubjectByCategory`).
- Reuses existing `getCategoryMeta`, `AMS_SUBJECTS`, `byCollection`-style
  HTML idioms.
- Naturally answers the most-asked question ("what's open vs solved in
  Number Theory?").
- No dependencies on git history, file scanning, or client JS.

### PR 2 — Solved/formal-proof ratios by field

**Deliverable:** a second section on `/stats/` ranking subjects (and
collections) by:
- formal-proof coverage (`hasFormalProof` rate within open+solved),
- solved rate (`research solved / (research open + research solved)`).

Two short tables, sorted descending, with N>=10 cutoff to avoid noise.
Useful for ATP benchmarking framing.

### PR 3 — Repo health metrics (low-risk infra)

**Deliverable:** a "Health" section listing:
- theorems missing `@[AMS]` (currently none should slip past the linter,
  but any orphan AMS code not in `AMS_SUBJECTS` should be flagged),
- theorems with `formal_proof` link that doesn't parse as URL,
- `module` strings whose collection segment is not in
  `SOURCE_COLLECTIONS` (they currently fall through to "Unknown"),
- count of `Other`-bucketed entries.

Each finding should be a list of `displayTheorem`s with file links.
This is build-side computation only — no Lean code or schema changes.

### PR 4 — Time evolution (git-history powered)

**Deliverable:** monthly time-series on `/stats/`:
- conjectures added per month,
- conjectures transitioning to `research solved` per month (best-effort
  via attribute diffs — see "git mining" below for caveats),
- formal-proof attributions added per month,
- contributors per month.

Render as either inline SVG sparklines (preferred — no JS deps) or a
small Chart.js include if we accept a dep (see "Open questions").

### PR 5 — Contributor analytics

**Deliverable:** "Contributors" section showing:
- top contributors by # of theorems first-added,
- top contributors by AMS area (heatmap or top-3 areas per contributor),
- most active source collections by recent commits.

This depends on the same git-mining pass as PR 4 — bundle them if PR 4
lands.

## 3. Data plumbing

### 3a. Reuse what's already in `conjectures.json`

Everything in PRs 1–3 can be computed from the already-loaded
`conjectures` array in `site/build.js`. No schema change, no Lean
changes, no new scripts. Add helpers next to `computeStats`:

```js
// site/build.js — extend computeStats or add computeAdvancedStats
function computeAdvancedStats(conjectures) {
  const subjectByCategory = {};   // { subjectName: { category: count } }
  const collectionByCategory = {};
  const subjectFormalCount = {};  // { subjectName: { withProof, total } }
  const collectionFormalCount = {};
  const health = {
    unknownCollection: [],   // displayTheorem list
    badFormalProofLink: [],
    unknownAmsCode: [],      // for any subject.code not in AMS_SUBJECTS
  };

  for (const c of conjectures) {
    // ... aggregate ...
  }
  return { subjectByCategory, collectionByCategory,
           subjectFormalCount, collectionFormalCount, health };
}
```

Write these into `site/data/conjectures.json` under a new top-level key
(e.g. `advancedStats`) so the client page can fetch the same file —
**don't duplicate the data**.

### 3b. Git mining for PR 4 / PR 5

Two viable strategies; pick one and stick to it.

**Option A — Build-time git scan (simpler, slower):**
Add `site/scripts/git_history.js` (or extend `build.js`) that runs
`git log --follow --diff-filter=A --pretty=format:'%H|%aI|%an|%ae'
-- FormalConjectures/**/*.lean` to get **first-add** date per Lean file,
joined to module name → conjectures. Output a `git_history.json`
artifact next to `conjectures.json`. Runs in a few seconds locally;
needs the full repo in CI (already true).

Tradeoffs:
- Captures **add** events accurately.
- Captures **modifications** (e.g. status change `open → solved`) only
  via a coarse pass: walk commits touching files containing
  `@[category` and diff the matched line. Inaccurate but useful as a
  signal; flag the methodology on the page.
- Contributor attribution uses `%an` — same caveats as `git shortlog`
  (aliases, bots).

**Option B — Lean-side history (cleaner, more work):**
Bake first-add date and last-status-change into the `extract_names`
output by reading git from the Lean script. Heavier — defer.

Recommend **Option A** for PR 4. Document the diff-based status
heuristic on the page itself.

### 3c. File-system scan (avoid)

Per the user's note ("you can scan FormalConjectures/**/*.lean"). We
**don't need to** for any of PRs 1–5: every conjecture in
`conjectures.json` already carries `module`, `category`, `subjects`,
`formalProofKind`. The Lean extractor is the source of truth; raw regex
on `.lean` files would risk drift. Health checks for "files not indexed"
can compare `glob(FormalConjectures/**/*.lean)` against the set of
`module`s seen in `conjectures.json` — that one cheap scan is the
exception.

## 4. Page wiring (concrete)

### 4a. Template

Create `site/src/templates/stats.html`. Structure mirrors `index.html`:
shared `<nav>`, `<header class="page-header">`, `<main>` with a series
of `<section class="section">` blocks, shared `<footer>`.

Server-rendered placeholders (filled by `build.js` via `fill()`):

```

{{ratiosTable}}            -- PR 2
{{healthSection}}          -- PR 3
{{timeSeriesSparklines}}   -- PR 4 (SVG inline)
{{contributorTables}}      -- PR 5
```

Keep all heavy data server-rendered — avoids needing client JS for the
basic page.

### 4b. `build.js` additions

1. Extend `computeStats` (or add `computeAdvancedStats`).
2. Add HTML generators next to `categoryStatsHTML` /
   `collectionListHTML` / `subjectListHTML`:
   - `subjectStatusTableHTML(subjectByCategory, byCategory)`
   - `ratiosTableHTML(...)`
   - `healthSectionHTML(health)`
3. Register the page in `main()`:

   ```js
   const statsHtml = readTemplate('stats.html');
   writePage('site/stats/index.html', applyBasePath(fill(statsHtml, {
     subjectStatusTable: subjectStatusTableHTML(...),
     ratiosTable: ratiosTableHTML(...),
     healthSection: healthSectionHTML(...),
     // PR4/5 placeholders later
   })));
   ```

4. Add a "Stats" entry to **every** template's `<ul class="site-nav__links">`
   (5 files). Cheap but easy to forget — checklist item.

### 4c. CSS

Add to `site/src/css/style.css`:
- `.stats-table` (sticky header, zebra rows, right-aligned counts),
- `.stats-table td.empty` (light gray for zeros),
- `.sparkline` (PR 4) — width/height, stroke color tokens.

Reuse existing `.badge.cat-*` classes for category cells; reuse
`.count-badge` for totals.

### 4d. URL filter contract

Browse page already accepts:
- `?category=research+open`
- `?collection=Wikipedia`
- `?subject=Number+theory`
- `?formal_proof=true` (used in `index.html:66`)

Verify these still work; the stats table cells should generate links
matching this contract. If the browse page can't combine
`?subject=...&category=...`, that's a `browse.js` follow-up — check
before promising drill-down on the stats page.

## 5. Failure modes and edge cases

- **Empty data locally.** `build.js` already exits if
  `conjectures.json` is missing. Stats generation must run *only after*
  that check; place all new code after line ~316.
- **AMS code for which no theorem exists.** Filter rows by `count > 0`
  before rendering — table would otherwise have ~80 mostly-empty rows.
- **Multi-AMS theorems.** A theorem with `@[AMS 5 11]` contributes to
  both rows; this is consistent with existing `bySubject` behavior in
  `computeStats`. Document this on the page so totals don't look like
  they don't sum — *row totals will exceed `conjectures.length`*.
- **Categories outside `CATEGORY_META`.** Anything new (`unknown`)
  surfaces as `cat-unknown`. The stats page should add it to the table
  rather than silently drop it; this also surfaces it as a health hint.
- **`formal_proof` without `category research solved`.** Allowed by the
  schema (proofs are independent). Don't conflate "solved" with "has
  formal proof" anywhere on the page; they are separate columns.
- **Git mining (PR 4) in CI.** Confirm the GitHub Actions checkout uses
  full history, not `fetch-depth: 1`. If shallow, either deepen the
  checkout for the stats build job or precompute the time series in a
  separate scheduled workflow that commits `git_history.json`.

## 6. Testing / verification

- Local: `cd site && npm run build` and open `site/site/stats/index.html`
  in a browser. (`site/site/` is the build output per `.gitignore`.)
- Spot-check totals: Number theory row total = grep count of
  `@[AMS 11` (or 11-containing) in the repo via the existing data. If
  off, it's almost certainly the multi-AMS double-count — see 5.
- View-source check: links in cells point at correct
  `/browse/?...` query strings (encodeURIComponent applied).
- Run on the actual data the CI uses: pull a recent
  `conjectures.json` artifact rather than relying on the placeholder
  one currently in `site/data/`.

## 7. Open questions for the user

These decisions affect PR scope and shouldn't be made unilaterally:

1. **Naming the URL.** `/stats/`, `/analytics/`, or `/insights/`?
   Suggest `/stats/` — most direct, matches the existing "stats-bar"
   class.
2. **Charts dep.** PR 4 renders cleaner with a tiny chart lib but the
   project explicitly avoids deps. Inline SVG sparklines are doable
   in vanilla JS — recommend that route.
3. **PR-1 scope.** Should it include the ratios from PR 2 (cheap to
   bundle), or strictly stay one section to keep the diff easy to
   review?
4. **Where to surface the page.** Add to top nav (visible everywhere,
   ~5 template edits) or only link from the home page initially?
5. **Anything Google-DeepMind-specific** to coordinate before
   contributing this externally? (The repo is `google-deepmind/`-owned
   — worth opening a "website feedback" issue describing the proposal
   before writing the PR.)

## 8. Out of scope (explicitly)

- New Lean attributes / metadata systems.
- Per-theorem analytics (page views, votes — `voting.js` is a separate
  concern).
- Server-side rendering or any backend.
- Dependency graphs between theorems (interesting but needs Lean-side
  dep extraction; defer indefinitely).
